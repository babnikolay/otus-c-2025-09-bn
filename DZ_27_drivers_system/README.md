# Проект LKM (Loadable Kernel Module): bn_device  
Данный проект содержит исходный код модуля ядра Linux (драйвер кольцевого буфера) и пользовательское 
приложение для его тестирования.  

Драйвер реализует потокобезопасную запись с использованием мьютексов и проверкой 
валидности указателей (NULL-pointer check), что исключает падение ядра (Kernel Panic) 
при одновременном обращении из разных интерфейсов (Device file и Sysfs).  

Реализована полная потокобезопасность (Thread-safety) для операций чтения и записи. 
Драйвер корректно обрабатывает конкурентный доступ между системными вызовами 
(User-space) и административными командами (Sysfs).  

## Структура проекта  
- **src/** — исходные файлы драйвера (module_ring.c) и теста (module_test.c).  
- **obj/** — временные объектные файлы (.o) пользовательского приложения.  
- **build/** — временные файлы сборки ядра (.ko, .mod, .cmd).  
- **Makefile** — основной сценарий сборки.  

## Особенности сборки  
Makefile настроен на строгую изоляцию:  
1. **Чистота src/:** Все временные файлы компиляции (артефакты Kbuild и объектные файлы GCC) 
выносятся в каталоги build/ и obj/.  
2. **User-space тест:** Собирается системным gcc с использованием std=c11.  
3. **Kernel-space модуль:** Собирается через Kbuild во внешней директории.  

## Основные команды  

### Сборка  
```sh
make
```

или  

```sh
make all
```

### Запуск тестов (автоматический цикл)  
Самая важная команда. Она компилирует код, загружает модуль, создает 
ноду в /dev, запускает тест, выводит логи ядра и выгружает модуль обратно:  
```sh
make test
```

### Управление модулем вручную  
__make load__ — скомпилировать, загрузить модуль (insmod) и создать файл устройства /dev/bn_device.  
__make unload__ — выгрузить модуль из памяти и удалить файл устройства.  
__make check_dev__ — проверить, существует ли файл устройства и корректно ли он определен в системе.  

### Установка в систему  
```sh
sudo make install    # Устанавливает модуль в системную директорию /lib/modules/...
make deploy          # Установка и немедленная загрузка через modprobe
```

### Очистка  
```sh
make clean      # Удалить все бинарные файлы, папки build/ и obj/
```

## Ручная проверка через sysfs  

### 1. Параметры модуля (Configuration)  
Драйвер поддерживает настройку размера кольцевого буфера при загрузке или через файловую систему sysfs.  

|   Параметр  | Тип |             Описание                    | По умолчанию |
|-------------|-----|-----------------------------------------|--------------|
| buffer_size | int | Общий размер кольцевого буфера в байтах |     1024     |

#### Примеры использования:  
1. Задание размера при загрузке:  
Установите буфер размером 4 КБ при ручной вставке модуля:  
```sh
sudo insmod build/bn_device.ko buffer_size=4096
```

2. Проверка текущего значения через sysfs:  
Посмотреть, какой размер буфера выделен в данный момент:  
```sh
cat /sys/module/bn_device/parameters/buffer_size
```

3. Изменение "на лету"  
```sh
echo 2048 | sudo tee /sys/module/bn_device/parameters/buffer_size
```

### 2. Проверка загрузки и версии  
Ядро автоматически создает записи для модуля в файловой системе /sys.  
Это позволяет проверять состояние драйвера без запуска тестового приложения.  

Убедитесь, что модуль виден в системе:  
```sh
ls -l /sys/module/bn_device
```

### 3. Проверка работоспособности модуля   
В одной консоли введите:  
```sh
cat /dev/bn_device
```

В другой консоли отправьте сообщение в буфер? например:  
```sh
sh -c 'echo "Hello lhfbblsabdfbakljfn;aksjnvfasdfjnsdfjna;sjfn;lasnfl" > /dev/bn_device'
```

Сообщение должно появиться в первой консоли.   

## Дополнение  
В каталоге также находятся модуль и тест из лекции.  
Чтобы проверить их работоспособность, необходимо в Makefile поменять цели MODULE_NAME, 
TARGET_TEST и SRC_FILENAME на соответствующие файлам модуля и теста.  
Файл Makefile_orig - базовый файл сборки для любого модуля без каких-либо зависимостей.
